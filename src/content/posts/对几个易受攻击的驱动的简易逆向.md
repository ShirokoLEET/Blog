**第一个：[ipctype.sys](https://github.com/lallouslab/vulnerable-drivers-ex/blob/main/drivers/ipctype.sys)**：
DriverEntry 反编译结果：

```c title="ipctype.sys: DriverEntry"

    RtlInitUnicodeString(&DestinationString, L"\\DosDevices\\IPCType");
```

这里的 `DriverObject->MajorFunction` 是驱动对象中的派发表，数组下标对应 IRP major function code。IDA 这里用十进制显示数组下标，因此：

| 下标 | 十六进制 | IRP major function | 含义 |
|---:|---:|---|---|
| `14` | `0x0E` | `IRP_MJ_DEVICE_CONTROL` | 处理用户态 `DeviceIoControl` 或上层驱动发来的 IOCTL 请求。 |
| `2` | `0x02` | `IRP_MJ_CLOSE` | 处理目标设备对象关联的文件对象最后一个句柄被关闭后的关闭请求。 |
| `0` | `0x00` | `IRP_MJ_CREATE` | 处理打开文件对象或设备对象句柄的请求。 |

也就是说，这个驱动把 `IRP_MJ_CREATE`、`IRP_MJ_CLOSE` 和 `IRP_MJ_DEVICE_CONTROL` 都交给同一个分发函数 `sub_110D0` 处理。其中最值得继续跟进的是 `IRP_MJ_DEVICE_CONTROL`，这个样本的敏感行为也落在 IOCTL 分发逻辑里。

继续逆向`sub_110D0`:

到这里可以确认，第二个参数就是传入进来的 IRP 值。这个函数先拿到 `Irp->Tail.Overlay.CurrentStackLocation`，再通过 `CurrentStackLocation->MajorFunction` 判断 IRP major function code；当值为 `14`，也就是 `IRP_MJ_DEVICE_CONTROL` 时，继续读取当前 I/O 栈里的 IOCTL 参数。

| 汇编偏移 | IDA 当前显示 | `Parameters.DeviceIoControl` 字段 | 用途 |
|---:|---|---|---|
| `+0x08` | `Parameters.Read.Length` | `OutputBufferLength` | 检查输出缓冲区长度。 |
| `+0x10` | `Parameters.Create.Options` | `InputBufferLength` | 检查输入缓冲区长度。 |
| `+0x18` | `Parameters.Read.ByteOffset.LowPart` | `IoControlCode` | 判断 IOCTL 控制码。 |

`sub_110D0` 中出现了四个 IOCTL 控制码，按 `CTL_CODE(DeviceType, Function, Method, Access)` 拆开如下：

| IOCTL | 有符号十进制显示 | DeviceType | Function | Method | Access | 行为 |
|---:|---:|---:|---:|---:|---:|---|
| `0x80102040` | `-2146426816` | `0x8010` | `0x810` | `METHOD_BUFFERED` | `FILE_ANY_ACCESS` | 调用 `sub_11270` 建立映射。 |
| `0x80102044` | `-2146426812` | `0x8010` | `0x811` | `METHOD_BUFFERED` | `FILE_ANY_ACCESS` | 调用 `sub_113B0` 释放映射。 |
| `0x80102048` | `-2146426808` | `0x8010` | `0x812` | `METHOD_BUFFERED` | `FILE_ANY_ACCESS` | 返回成功，`IoStatus.Information` 保持为 `0`。 |
| `0x8010204C` | `-2146426804` | `0x8010` | `0x813` | `METHOD_BUFFERED` | `FILE_ANY_ACCESS` | 返回成功，`IoStatus.Information` 保持为 `0`。 |

因为四个 IOCTL 的 `Method` 都是 `0`，也就是 `METHOD_BUFFERED`，所以用户态传入的缓冲区会被 I/O 管理器放到 `Irp->AssociatedIrp.SystemBuffer`。汇编里 `mov rsi, [rdx+18h]` 对应的就是这个缓冲区地址。

`0x80102040` 会根据调用进程是否为 32 位进程走两套长度检查和字段偏移：

| 进程类型 | `InputBufferLength` | `OutputBufferLength` | 访问偏移 | 行为 |
|---|---:|---:|---|---|
| 32 位进程 | `0x10` | `0x10` | 读取 `[SystemBuffer+0x04]` | 作为 `NumberOfBytes` 传入 `sub_11270`。 |
| 32 位进程 | `0x10` | `0x10` | 读取 `[SystemBuffer+0x08]` | 使用 `movsxd` 取出 32 位物理地址并符号扩展后传入 `sub_11270`。 |
| 32 位进程 | `0x10` | `0x10` | 写入 `[SystemBuffer+0x0C]` | 写回 `sub_11270` 返回的映射地址低 32 位。 |
| 64 位进程 | `0x1C` | `0x1C` | 读取 `[SystemBuffer+0x08]` | 作为 `NumberOfBytes` 传入 `sub_11270`。 |
| 64 位进程 | `0x1C` | `0x1C` | 读取 `[SystemBuffer+0x0C]` | 取出 64 位物理地址后传入 `sub_11270`。 |
| 64 位进程 | `0x1C` | `0x1C` | 写入 `[SystemBuffer+0x14]` | 写回 `sub_11270` 返回的 64 位映射地址。 |

`0x80102044` 是对应的释放接口：

| 进程类型 | `InputBufferLength` | `OutputBufferLength` | 访问偏移 | 行为 |
|---|---:|---:|---|---|
| 32 位进程 | `0x00` | `0x10` | 读取 `[SystemBuffer+0x0C]` | 作为参数传入 `sub_113B0`。 |
| 64 位进程 | `0x00` | `0x1C` | 读取 `[SystemBuffer+0x14]` | 作为参数传入 `sub_113B0`。 |

长度检查不满足时，`sub_110D0` 会把 `Irp->IoStatus.Status` 设置为 `0xC000000D`，也就是 `STATUS_INVALID_PARAMETER`。未知 IOCTL 也走同一个状态码。处理结束后统一调用 `IofCompleteRequest(Irp, 0)` 完成 IRP。

继续看 `0x80102040` 调用的 `sub_11270`：

```c title="ipctype.sys: sub_11270"
__int64 __fastcall sub_11270(PHYSICAL_ADDRESS MdlAddress_low, unsigned int NumberOfBytes, _QWORD *a3)
{
  SIZE_T Length; // r12
  unsigned int v5; // ebx
  PVOID VirtualAddress; // rax
  void *BaseAddress; // rsi
  struct _MDL *Mdl; // rax
  struct _MDL *Mdl_1; // r13
  PVOID v10; // rdi
  _QWORD *PoolWithTag; // rax

  Length = NumberOfBytes;
  v5 = 0;
  VirtualAddress = MmMapIoSpace(MdlAddress_low, NumberOfBytes, MmNonCached);
  BaseAddress = VirtualAddress;
  if ( VirtualAddress )
  {
    Mdl = IoAllocateMdl(VirtualAddress, Length, 0, 0, 0);
    Mdl_1 = Mdl;
    if ( Mdl )
    {
      MmBuildMdlForNonPagedPool(Mdl);
      v10 = MmMapLockedPagesSpecifyCache(Mdl_1, 1, MmNonCached, 0, 0, 0x10u);
      if ( v10 )
      {
        PoolWithTag = ExAllocatePoolWithTag(NonPagedPool, 0x28u, 0x54435049u);
        PoolWithTag[1] = Mdl_1;
        PoolWithTag[2] = BaseAddress;
        PoolWithTag[3] = v10;
        *((_DWORD *)PoolWithTag + 8) = Length;
        *PoolWithTag = P;
        P = PoolWithTag;
        *a3 = v10;
      }
      else
      {
        MmUnmapIoSpace(BaseAddress, (unsigned int)Length);
        return (unsigned int)-1073741670;
      }
    }
    else
    {
      MmUnmapIoSpace(BaseAddress, Length);
      return (unsigned int)-1073741670;
    }
  }
  else
  {
    return (unsigned int)-1073741670;
  }
  return v5;
}
```

这段逻辑的执行顺序非常直接：

1. 使用用户提供的 `PhysicalAddress` 和 `NumberOfBytes` 调用 `MmMapIoSpace`，把物理地址范围映射到内核虚拟地址。
2. 对 `MmMapIoSpace` 返回的 `BaseAddress` 调用 `IoAllocateMdl`。
3. 调用 `MmBuildMdlForNonPagedPool` 初始化 MDL。
4. 调用 `MmMapLockedPagesSpecifyCache`，并且第二个参数传入 `UserMode`，把同一段页映射到用户态地址空间。
5. 使用 `ExAllocatePoolWithTag(NonPagedPool, 0x28, 0x54435049)` 分配一条记录，插入全局链表 `P`。
6. 通过第三个参数把用户态映射地址写回给调用方。

全局链表 `P` 的每条记录大小为 `0x28`，字段由写入位置确认如下：

| 偏移 | 内容 | 写入位置 |
|---:|---|---|
| `0x00` | 下一条记录指针 | `*PoolWithTag = P` |
| `0x08` | `Mdl` | `PoolWithTag[1] = Mdl` |
| `0x10` | `BaseAddress` | `PoolWithTag[2] = BaseAddress` |
| `0x18` | `UserAddress` | `PoolWithTag[3] = UserAddress` |
| `0x20` | `Length` | `*((_DWORD *)PoolWithTag + 8) = Length` |

这里的核心问题是：调用方可以控制传入 `MmMapIoSpace` 的物理地址和长度，驱动随后又把映射结果通过 `MmMapLockedPagesSpecifyCache(..., UserMode, ...)` 暴露到用户态。驱动没有对物理地址范围做限制，也没有检查调用方权限；因此用户态可以借这个 IOCTL 映射任意物理地址范围。

再看 `0x80102044` 调用的 `sub_113B0`：

```c title="ipctype.sys: sub_113B0"
__int64 __fastcall sub_113B0(PVOID MdlAddress_high)
{
  PVOID *P; // rbx
  PVOID *P_1; // rdi

  P = (PVOID *)P;
  P_1 = (PVOID *)P;
  while ( P )
  {
    if ( P[3] == MdlAddress_high )
    {
      MmUnmapLockedPages(P[3], (PMDL)P[1]);
      IoFreeMdl((PMDL)P[1]);
      MmUnmapIoSpace(P[2], *((unsigned int *)P + 8));
      if ( P == P )
        P = *P;
      else
        *P_1 = *P;
      ExFreePool(P);
      return 0;
    }
    P_1 = P;
    P = (PVOID *)*P;
  }
  return 0;
}
```

IDA 的反编译输出里，`sub_113B0` 的局部变量名与全局链表头 `P` 同名；链表更新语义以汇编里的全局写入为准，涉及 `0x11401` 处读取全局 `P`、`0x1140A` 处写回全局 `P`。

这个函数遍历全局链表 `P`，查找偏移 `0x18` 保存的用户态映射地址是否等于传入参数。找到后依次执行：

1. `MmUnmapLockedPages(Current[3], (PMDL)Current[1])`
2. `IoFreeMdl((PMDL)Current[1])`
3. `MmUnmapIoSpace(Current[2], *((unsigned int *)Current + 8))`
4. 从链表 `P` 中摘除当前记录
5. `ExFreePool(Current)`

如果没有找到匹配记录，函数仍然返回 `0`。这一点会让调用方无法通过返回值区分“释放成功”和“没有找到记录”。

最后整理一下 `ipctype.sys` 的攻击面：

| 位置 | 结论 |
|---|---|
| 设备名 | `\\Device\\IPCType` |
| 符号链接 | `\\DosDevices\\IPCType` |
| 设备类型 | `0x8010` |
| 关键 IOCTL | `0x80102040`、`0x80102044` |
| 映射函数 | `sub_11270` |
| 释放函数 | `sub_113B0` |
| 全局链表头 | `P`，地址 `0x13110` |
| 主要风险 | 用户态可控制物理地址和长度，并拿到用户态映射地址。 |
| 额外风险 | `ExAllocatePoolWithTag` 返回值未判空；全局链表 `P` 的增删路径没有同步保护。 |
